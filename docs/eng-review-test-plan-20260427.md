## Affected Pages/Routes

- `/strategies/marketplace` — prospect's first surface; must render the curated forex strategy with description, agent breakdown, "Import" button
- `/strategies/:id` (strategy detail) — full strategy view before import; agent reasoning preview, risk notes, target market
- `/my-strategies` — post-import view; prospect sees imported strategy ready to activate
- `/pipelines` — pipeline list; imported strategy materializes as an inactive pipeline
- `/pipelines/:id` (pipeline detail) — activation toggle, schedule config, broker selection (must default to OANDA paper)
- `/monitoring` — live execution view during demo; agent activity must be visible
- `/monitoring/execution-report-modal` — rich agent reasoning report; the "where confidence happens" surface
- `/auth/login` — per-prospect demo accounts use existing JWT login flow

## Key Interactions to Verify

### On `/strategies/marketplace`
- Marketplace card renders for the curated forex strategy (visibility=public, publication_status=published)
- "Import" CTA is visible and clickable for non-owner users
- Strategy detail link navigates to `/strategies/:id`

### On `/strategies/:id`
- Markdown content renders (description, entry rules, exit rules, risk notes)
- "Import to my account" creates a Pipeline row in prospect's account
- Post-import navigation lands on `/my-strategies` or `/pipelines`

### On `/pipelines/:id`
- "Activate" toggle starts the signal-generator → trigger-dispatcher flow for this pipeline
- Broker selection defaults to OANDA paper (`account_type == "demo"`)
- Approval flag works end-to-end if enabled

### On `/monitoring`
- Active pipelines visible in real-time
- Agent activity stream (which agent is running, decision in progress) is visible during a signal trigger
- Execution row appears when a signal fires

### On `execution-report-modal`
- Each agent's report renders with: agent name, decision, reasoning, confidence
- Multi-agent disagreement is visible (where one agent says yes, another says no)
- Final decision and broker order outcome render

## Edge Cases

### Marketplace
- Strategy with empty `markdown_content` — does it render or break?
- Strategy import for already-imported strategy — duplicate or no-op?
- Non-authenticated user lands on marketplace — redirect to login or anonymous browse?

### Pipeline activation
- Activating without OANDA broker connected — error state must be clear
- Activating outside forex session hours (Sat-Sun) — pipeline accepts but no signals fire; UI should communicate this clearly
- Multiple prospects activating the SAME imported strategy concurrently — each runs in their own account; no cross-talk

### Signal flow
- signal-generator produces a signal → Redis Streams xadd succeeds → trigger-dispatcher xreadgroup reads it → matches to active pipeline → enqueues Celery task → execution starts → agents run → report writes to DB → UI updates via WebSocket
- Redis down mid-publish: publish retries, eventually drops with telemetry counter increment
- trigger-dispatcher dead during xadd: signal sits in stream; on dispatcher restart, xreadgroup picks up pending entries (no message loss within Redis AOF window)
- Agent failure mid-pipeline: error reported in execution.reports; no broker order placed
- OANDA paper API rate limit hit: trade execution fails gracefully; reported in execution surface

### Plain-English authoring (StrategyAgent)
- Vague English ("buy when bullish") — agent should request clarification or produce a sensible default
- Contradictory rules ("only longs, but go short on stop loss") — agent should flag the contradiction
- Multi-leg options — agent must produce structurally valid output
- Non-English input — graceful failure with clear error
- Extremely long input (>10K chars) — truncation or rejection path

## Critical Paths

### Demo path (golden E2E)
1. Provision per-prospect demo account via admin UI (OR use a test account)
2. Prospect logs in
3. Lands on `/strategies/marketplace`
4. Clicks the curated forex strategy
5. Reviews strategy detail page
6. Clicks "Import"
7. Lands on `/my-strategies`, sees imported strategy
8. Navigates to `/pipelines`, sees the imported strategy as an inactive pipeline
9. Activates the pipeline (OANDA paper broker)
10. Forex market is open (Sun 5pm ET → Fri 5pm ET); within ~5 minutes a signal triggers
11. Multi-agent reasoning runs (visible on `/monitoring`)
12. Trade fires on OANDA paper account
13. Execution report renders in `execution-report-modal` with agent decisions

### Founder authoring path
1. Founder logs in
2. Creates a new strategy via plain-English description in StrategyAgent UI
3. StrategyAgent produces a runnable strategy (validates against expected schema)
4. Founder reviews, edits if needed, marks visibility=public, publication_status=published
5. Strategy appears in the marketplace for prospects

### Per-prospect provisioning path
1. Founder uses existing user-management to create demo account before each Zoom call
2. Account is sandboxed (own Pipelines, own Executions, no cross-talk)
3. Account credentials are shared with the prospect

### Bus reliability (Redis Streams)
1. signal-generator publishes 10 signals in 10 seconds
2. trigger-dispatcher consumes all 10, acks all 10
3. Restart trigger-dispatcher; verify no duplicate processing of acked entries
4. Restart trigger-dispatcher with 3 unacked pending entries; verify recovery via XPENDING
5. Configure Redis with `appendonly yes` `appendfsync everysec`; kill Redis hard; verify <1s of message loss

## LLM Eval (Issue 3C)

File: `backend/tests/evals/test_strategy_authoring.py` (new)

Marker: `@pytest.mark.accuracy` (existing; excluded from fast test runs)

Cases (5-10):
1. Simple RSI strategy: "Long EUR/USD when RSI(14) below 30, exit when above 70" → expect entry/exit conditions, position sizing, RSI(14) in indicators
2. Multi-condition: "Long when RSI oversold AND VIX below 25 AND it's London session" → expect AND-composed conditions
3. Risk explicit: "Max 1% account risk per trade" → expect position_sizing.max_risk_per_trade == 0.01
4. Stop loss + take profit: "5% stop loss, 10% take profit" → expect SL/TP in exit conditions
5. Vague: "buy when bullish" → expect either clarification request or sensible default with warning
6. Contradictory: "only longs but short on signal X" → expect contradiction flag
7. Forex-specific: "Long EUR/USD on London open breakout" → expect session-aware logic
8. Long-form: ~500 word description with multiple rules → expect well-structured extraction

Each case asserts: structural validity (JSON schema), key fields present, no hallucinated indicators (fail if agent references nonexistent indicator).

## Regression Tests (CRITICAL — non-negotiable per skill iron rule)

### `signal-generator/tests/test_signal_emission.py` rewrite
Currently mocks `KafkaProducer`. After Redis Streams swap, this test must:
- Mock `redis.Redis.xadd` instead
- Assert correct stream key (`trading-signals`)
- Assert MAXLEN cap is applied
- Assert telemetry counters increment (`stream_publish_success_total`, `stream_publish_failure_total`)
- Assert retry on `redis.exceptions.ConnectionError`

### NEW: `trigger-dispatcher/tests/test_consumer.py`
This file does not exist today. trigger-dispatcher has zero test coverage. The Redis Streams swap is the forcing function to introduce the first tests:
- `test_consumer_group_creation_idempotent` — calling create-group twice returns success
- `test_xreadgroup_happy_path` — receives entry, calls match_signals_to_pipelines, enqueues Celery, calls xack
- `test_xreadgroup_empty_poll` — no message in BLOCK 100 window, loop continues
- `test_xack_only_after_celery_success` — Celery enqueue fails, xack is NOT called, entry remains pending
- `test_pending_entry_recovery_on_restart` — restart consumer with 3 pending entries, all processed
- `test_telemetry_increments` — stream_consume_*, stream_ack_* counters fire correctly

### NEW: `trigger-dispatcher/tests/conftest.py`
Pytest fixture providing a mock Redis client matching backend's existing test patterns.
