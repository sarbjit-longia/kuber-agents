## Problem Statement

kuber-agents is a pre-product AI-agent-based trading platform where retail traders visually connect agents to build automated strategies. The codebase is mature (Angular 17 + FastAPI + Celery + Kafka + multi-broker integrations + langfuse tracing + agent registry), recently migrated from CrewAI to OpenAI agents, with no users yet.

The session surfaced two unvalidated load-bearing assumptions: (1) a path to reach 1,000 retail traders, and (2) demand evidence beyond founder conviction. The product wedge as originally framed ("AI agents that make strategies dynamic") is also crowded as of 2026 (Public.com, Trader.ai, WunderTrading, DeepSeek+Alpaca, HKUDS AI-Trader, TradingAgents framework), so re-framing to a more defensible wedge was a key outcome.

## Demand Evidence

Founder claimed "someone keeps asking when it ships" and "a specific named trader I can quote." Under direct push for the literal name, role, and quote, no concrete evidence was produced. The honest read: demand evidence is **founder conviction + a small social circle of trading buddies + a composite user who likely overlaps with the founder's own profile**.

This is a fine place to be IF it is faced. It is dangerous if it is treated as validated.

## Status Quo

Target user's current workaround is **existing automation written in code or a DSL**: TradingView Pine Script with webhooks to brokers, 3Commas-style bots, or custom Python scripts wired into broker APIs. The specific pain articulated in their words: "They use multiple systems and it's brittle. Also it does not support using AI agents in their strategy. Strategy is hardcoded using code rather than dynamic based on various factors."

The implication for kuber-agents' wedge: traders are *not* migrating from TradingView by importing their webhook — they are **escaping the code-first authoring model entirely**. kuber-agents replaces "learn Pine Script / write Python / wire webhooks" with "describe what you want in English; agents handle authoring, signal detection, reasoning, and execution." The status quo isn't just brittle — it requires the user to *be* the strategy programmer. kuber-agents removes that requirement.

## Target User & Narrowest Wedge

**Target archetype:** Side-hustle trader with a tech day job, account size $25K–$150K, trades options/futures evenings/early mornings, dreams of quitting their job to trade full-time but waits for multi-year proof. Time-poor, money-OK, technically fluent, has already crossed the automation chasm.

**Narrowest wedge (founder-revised after pushback on Codex's narrowing):** A **multi-agent reasoning platform with rich reports as the trust-building surface** — *not* a stripped 3-block linear pipeline.

The founder pushed back on Codex's strip-to-3-blocks recommendation with a stronger product insight: agent reasoning visibility is where user confidence is built, and the existing architecture (signal generator → dispatcher → multiple agents → reports) is a richer reasoning surface than 3 deterministic blocks. The wedge becomes **transparent multi-agent reasoning**, not minimal control plane.

**Architecture scope (keep rich):**
- Signal generator, dispatcher, full agent registry stay
- Multiple specialized agents (analysis, risk, execution, reporting) collaborate on each decision
- AI report agent stays — and gets *richer* (this is the moat against Public.com / Trader.ai which lean on autonomous portfolio agents, not transparent reasoning)

**Demo scope (still narrow, with reports as centerpiece):**

The killer flow for cold prospects (90-second video / Zoom demo) — using forex (OANDA paper) for 24/7 market access so demos don't depend on US stock hours:

1. Prospect lands in the **strategy marketplace** (existing UI), browses one curated forex strategy authored by the founder.
2. Prospect imports the strategy into their account (existing one-click import flow). Sees the strategy described in plain English in a strategy-authoring agent. Example: *"Long EUR/USD when RSI(14) below 30 and the London session open is within 1 hour. Max 1% account risk per trade. Exit on RSI above 70 or 1.5% stop loss."*
3. Prospect activates the pipeline. Internal signal-generator monitors live OANDA EUR/USD data; when conditions trigger, agents reason about the proposed trade in real time (visible agent activity in the monitoring UI, not a black box).
4. **Rich report** (existing `execution-report-modal`) explains what each agent concluded, where they agreed, where they disagreed, what the final decision was, and *why*.
5. Paper trade fires on OANDA if approved by the risk gate. Replay of recent signals, each with its own full reasoning trail.

For prospects who want to author their own: same flow, except step 1 is "skip the marketplace, write the strategy in plain English directly." The marketplace import path is the *bait*; the plain-English authoring is the *unlock*.

Reports become *the* feature the prospect remembers — not "one feature among many." This is a stronger demo than a TradingView-import narrowing, because (a) plain-English authoring removes the Pine Script learning curve entirely, and (b) the multi-agent reasoning trail is the trust-building surface that "import your webhook" tools cannot match.

**The category framing this implies:** kuber-agents is *not* "TradingView/3Commas with AI." It is a **natural-language strategy authoring platform** where AI agents translate English into runnable, auditable, multi-agent-reasoned strategies. That's a different category, harder to compare against existing tools, easier to explain in 30 seconds to a non-technical retail trader.

**What gets deferred from the demo (not deleted from code):**
- Agent marketplace UI / agent picker — hardcode one curated config for V1 prospects
- Multi-broker selection — point at one paper broker (Alpaca paper)
- Scanner mode — use one curated symbol set
- Full pipeline builder UI complexity — preset workflow for V1 prospects; let curious traders see the builder after they're hooked

This is *not* "AI invents trades." It is "watch the agents reason, see the report, decide whether to trust it." The wedge is **transparent multi-agent reasoning** built on the *control plane infrastructure you already have* (langfuse, approvals, cost tracking).

## Constraints

- **Solo-founder (or small team)** — bandwidth is the bottleneck, not capital
- **No external user validation yet** — this is the binding constraint, not technical capability
- **Existing 21-service stack** has been built broadly, must be narrowed sharply
- **Cost-conscious** — explicit founder concern that infra burn before revenue is a risk
- **Legal posture** for retail trading tools must be handled before real money flows

## Approaches Considered

### Approach A: Strategy Ops Console for Retail (Codex's narrowing) — *superseded by founder revision; rolled into Approach D Phase 3*
Codex's original prescription was to strip to 3 deterministic blocks (signal filter → AI explainer → risk gate) and import TradingView webhook payloads. Founder pushed back: the existing multi-agent architecture and rich reasoning reports are the moat, and the actual product is **plain-English strategy authoring**, not webhook import.

This approach is **superseded by the revised demo flow in "Target User & Narrowest Wedge"** (plain-English authoring → multi-agent reasoning → rich reports) and rolled into Approach D Phase 3 as the build target. It is not a live alternative on its own.

### Approach B: Strategy Ops Console for B2B
Same product, sold to prop firms (FTMO-style), small RIAs, family offices. Compliance teams *already* want governance + audit + kill switches + approval workflows. Distribution problem becomes "find 50 firms" instead of "find 1,000 retail traders." Higher LTV ($5K-$50K/yr), longer sales cycles. Risk: low-medium.

### Approach C: No-Build Discovery Sprint
30 days, zero new product code. Talk to 20 arms-length retail algo traders (not friends). Use interview script "walk me through the last time you wanted to change your live strategy and didn't." Pre-sell to 5 with $99 deposits as design partners. Build only after deposits land. Risk: low.

Founder pushed back on Approach C with "I need something to show" — legitimate concern about demoless discovery. Approach D resolves the tension:

Configure Redis with `appendonly yes` `appendfsync everysec` for at-least-once durability (accepts up to 1s loss on crash). Direct rewrite, no abstraction layer. Drop `kafka` + `zookeeper` services. Final footprint: **10 services / ~4 GB**.

  Cross-model note: two AI reviewers independently flagged this swap as premature optimization for a 10-day sprint. Founder retained the original choice based on context not visible to reviewers (preference for cleaner production footprint, plan to scale Redis path post-Phase-3). Acknowledged risk: 1-2 days that could otherwise go to demo quality work (prompt iteration, signal-during-demo problem).

Run the stripped `docker-compose.prod.yml` directly on the VM. Public HTTPS via Caddy reverse proxy or your existing nginx setup.

## Legal Posture for Retail Trading Tool

Must be in place before paid users:

- **Frame as a tool, not an advisor.** "Your strategy, our execution." Never "our AI picks winners."
- **No custody, ever.** Users connect their own broker accounts (already supported via Tradier/Alpaca/OANDA integrations). You never hold user funds. This avoids most MSB / money transmitter law.
- **Paper trading default.** First N days or trades on paper before live execution is unlocked.
- **Disclaimers, prominently placed.** "Not investment advice. Past performance not indicative. Trading involves substantial risk of loss." Must appear on every page touching strategy/trades.
- **Strong ToS** with arbitration clause and liability waiver. Start from a Stripe Atlas template; do **not** treat it as final.
- **Geographic gating** in early stages. If not US-registered, restrict to specific jurisdictions.
- **No backtest-result claims in marketing** without proper hypothetical-performance footnotes (SEC is aggressive here).
- **Commit to one fintech lawyer consultation, $1.5K budget, BEFORE Phase 2 outreach starts (revised after cross-model review).** Originally scheduled before Phase 3 build, but the $99 non-refundable deposit creates a regulatory exposure as soon as Phase 2 collects money — refund disputes, jurisdictional taxonomy of the deposit, and ToS specifics need legal sign-off *before* any prospect pays. Bring the ToS template, the structure (no custody, paper-default, broker-connect model), the AI explainer description, the deposit terms, and the geographic gating question (US-only / US-excluded / specific countries). Get a written memo. This is a hard gate, not a nice-to-have.

## Phase 1 design decisions added during /plan-design-review

These items were surfaced during `/plan-design-review` on 2026-04-27 and added to Phase 1 scope. All target the demo conversion path through existing UI surfaces.

- **`execution-report-modal`: Add Agent Consensus pill at top** — small chip after Executive Summary showing `[N]/[Total] agents agree: [decision]` with optional disagreement count. Visual: small chip near top with semantic color (green for full agreement, amber for partial). Backend must compute consensus from `agent_reports`. Sets the multi-agent narrative within the first 5 seconds of reading the modal.
- **`execution-report-modal`: Empty-state fallback** — when `executiveReport` is falsy, `loading` is false, and `error` is null/empty, render: *"Report being generated. This usually takes 5–15 seconds. Refresh in a moment."* with a small refresh button. Prevents the demo-killing empty-modal moment.
- **`execution-report-modal`: Show errored agents in the accordion** — failed agents render as expansion panels with a warning icon and the error message inside. They count toward the consensus pill (e.g., "3/5 agents reported, 2 errored"). Maintains the transparency-as-wedge thesis even on partial failure.
- **`execution-report-modal`: Detailed Agent Analysis stays collapsed by default** — explicit decision (against initial recommendation). Founder will watch the first 3 demos for whether prospects click into agent panels at all; if they don't, revisit auto-expansion.
- **`monitoring` view: Streaming agent activity feed during pipeline execution** — between "signal fires" and "report visible" (5–15s), monitoring view shows a live activity feed: agent name + decision verb + latency, animated entry as each agent completes. Reuses existing WebSocket. Backend must emit per-agent events; frontend renders the feed. Without this, the "watch agents reason" moment deflates into a spinner.
- **`my-strategies` page: Onboarding empty state** — first authenticated screen for prospect after login. Replace generic empty list with: hero illustration/icon + *"Welcome. Start by browsing strategies in the marketplace."* + prominent **Browse Marketplace** CTA + small text *"Or describe your own strategy in plain English"* linking to authoring. Empty state becomes a moment of orientation.

These six items extend Phase 1 scope by an estimated 1–2 days of frontend work plus minor backend changes (per-agent execution events, consensus computation, empty-state component). Budget the additional time accordingly.
